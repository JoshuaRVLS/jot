# src/

## Purpose

C++ engine source for `jot`. Hosts the editor's modular static-library layout, the bundled Python theme runtime support, and platform-specific shims.

## Ownership

- Owns: all C++ engine code, the embedded Python theme runtime scripts, and platform shims.
- Does not own: the executable entrypoint (`apps/jot/`), public C++ API headers (`include/`), tests (`tests/`), or user-facing docs (`docs/`).

## Local Contracts

- Module layout is fixed by `src/CMakeLists.txt` plus each module's `sources.cmake`. Each top-level subdirectory is one CMake `OBJECT` library and a matching `STATIC` library of the same name (`jot_core`, `jot_edit`, `jot_features`, `jot_input`, `jot_render`, `jot_tools`, `jot_python_bridge`, `jot_ui`).
- `jot_engine` is the aggregated static library consumed by the `jot` executable. Do not add sources directly to `jot_engine`; add them to their module `OBJECT` library and re-aggregate.
- `jot_configure_object_target` and `jot_expose_module_api` in `src/CMakeLists.txt` define the include and Python flags applied to every module. New modules must use both. New source files must be added to the owning module's `sources.cmake`; keep `src/CMakeLists.txt` focused on target wiring and dependencies. File names should not repeat their parent module or domain folder (`feature_*`, `render_*`, `command_*`, `mode_*`, `tree_sitter_*`); use concise snake_case nouns/actions and add a focused subfolder when the domain name is still needed.
- `src/CMakeLists.txt` requires pkg-config packages `vterm`, `termkey`, and `libuv`; libutf8proc is also required and is discovered through `libutf8proc`/`utf8proc` pkg-config names or a CMake header/library fallback. `jot_tools` owns the libvterm include flags for integrated-terminal emulation, `jot_ui` owns libtermkey and libutf8proc include flags for editor keyboard decoding and Unicode text handling, and `jot_core` owns libuv include flags for the editor event loop. `jot_engine` propagates all four link dependencies.
- Tree-sitter is optional parser-based syntax highlighting. `JOT_TREESITTER=ON` links only the `tree-sitter` runtime; parser grammars are discovered and loaded at runtime from shared libraries via `features/tree_sitter/manager.cpp` and `features/tree_sitter/runtime.cpp`. Missing or incompatible grammars fall back to the regex highlighter. `features/tree_sitter/languages/` owns per-language descriptors for catalog metadata and built-in highlight queries; `features/tree_sitter/catalog.*` adapts those descriptors to the catalog API, and `features/tree_sitter/install.*` owns `:tsinstall` source-build commands for user-local parser libraries and highlight query copies. Tree-sitter capture names map to named token kinds first, then render through theme slots; keep new token kinds backward compatible by defaulting them to existing palette families.
- Platform shims are not currently active. The supported engine sources are POSIX-only, and the root `CMakeLists.txt` blocks native Windows at configure time.
- Python runtime scripts under `src/python/` are installed as data (`${CMAKE_INSTALL_DATADIR}/jot/python`); do not move them to a `data/` folder and do not exclude `__pycache__`/`.pyc` from cleanup.

### Module responsibilities

- `core/`: editor state, buffers, panes, workspace, libuv-backed event loop, task queue, popup, home menu, theme, git, bookmarks, undo, LSP wiring, debugger session orchestration, integrated-terminal hosting. The structural heart of the editor. Stable internal headers such as `editor.h`, `editor_models.h`, `editor_state.h`, `types.h`, `event_loop.h`, line-provider headers, `task_queue.h`, and `host_api.h` stay directly under `src/core/`; implementation files live in focused subfolders:
  - `core/app/`: editor lifecycle, file/buffer/pane mechanics, undo, bookmarks, folding glue, utility methods.
  - `core/workspace/`: workspace sessions, file tree, git, task execution.
  - `core/integrations/`: LSP orchestration, debugger orchestration, integrated-terminal editor glue, editor syntax cache/highlighting glue, Tree-sitter commands.
  - `core/runtime/`: libuv event loop, task queue, line-provider implementations.
  - `core/surfaces/`: home screen, popup, quick-pick, theme, host API bridge.
  `editor.h` is the `Editor` facade and method declaration surface. Keep internal model structs/enums in `editor_models.h` and editor storage in `editor_state.h`; do not move large state blocks back into `editor.h`.
  **Renderer contract — paintable width vs physical width:** The renderer intentionally does not write the rightmost physical column (fixed at exactly one cell) to avoid the terminal's pending-wrap state at large widths. Wider margins are not supported because they create an oversized blank strip on the right edge. `UI::get_width()` returns the full physical terminal width; `UI::get_render_width()` returns `width - render_margin()`, the actually paintable area. Any layout that places a visible right edge (pane borders, status line, integrated terminal panel, image viewer, home menu background, telescope) **must** use `get_render_width()`. Layout that does its own coordinate clamping (mouse click handlers, centered popups, cursor positions) can use `get_width()` since those are soft. `UI::draw_border()` defensively clamps the requested right edge into the paintable area so future full-width panels cannot silently lose their right border.
  **Undo:** `Editor::save_state()` captures a `State` delta (not a full snapshot) so typing on large files (5M+ lines) is O(window) instead of O(file). For buffers with ≤ `kMaxFullSnapshotLines` (5000), the delta stores every line as a full snapshot for simplicity. For larger buffers, the delta stores only `kDeltaWindowHalfSize` (50) lines above and below the cursor, or the full selection range if one is active, plus cursor/selection/scroll/modified state. `save_state()` never calls `materialize()`, so the lazy provider remains valid through edits. Undo/redo apply the delta by computing the difference between the current line count and the captured `old_total_lines` to determine how many lines the edit inserted/deleted, then replacing the affected range with `old_lines` via `FileBuffer::replace_lines()`. `same_state()` compares the captured window and cursor state, never the full file, so consecutive identical undo entries are deduplicated in O(window) instead of O(n). The legacy "save_state copies all lines" pattern would freeze the editor for seconds on a 5M-line file because every keystroke allocated ~418 MB for the snapshot.
- `ui/`: raw terminal abstraction, full-row paint renderer, editor stdin decoding via libtermkey, and internal reusable UI primitives. All escape-sequence I/O for the main editor UI flows through here. `text.*` owns libutf8proc-backed UTF-8 validation, NFC/NFD normalization, grapheme-ish cursor boundaries, and terminal-cell measurement/truncation helpers. `components.*` owns small drawing helpers for repeated panels, titles, footers, and selectable rows; it is internal to jot and not a public widget framework. The `Terminal` class owns the cached `width`/`height`; `Terminal::refresh_size(bool force_probe = false)` re-probes the OS (ioctl → `/dev/tty` → `$COLUMNS`/`$LINES`) and is the canonical way to pick up a size change outside a `SIGWINCH`. When `force_probe` is `true` (or ioctl fails), it also runs an ANSI cursor-position probe (`CSI 999;999H` + `CSI 6 n`) that reads the real rows/cols from the terminal's response. `Terminal::init()` calls `refresh_size(/*force_probe=*/true)` once after entering the alternate screen and raw mode, before the first `UI::resize(...)`. Callers that resize the UI later must also call `ui->resize(...)`, `update_pane_layout()`, `ui->invalidate()`, and set `needs_redraw = true` so the layout, pane heights, and the diff renderer all pick up the new size atomically.
  **Renderer:** `UI::render()` paints every row from column 0 with explicit colors and pads rows through the safe render width — no diff against `last_grid`. Autowrap is disabled (`CSI ?7 l`) for frame and cursor output, and is restored only during terminal cleanup (`CSI ?7 h`). The renderer must not write the rightmost physical column of **any** row: writing the rightmost cell leaves the terminal in a pending-wrap state and at large widths the next cursor move can be misinterpreted as a wrap and scroll the viewport. The fixed one-cell margin keeps the visible right border on the last paintable column instead of a wider blank strip. After writing each row the renderer emits `EL 0` (`\x1b[K`, erase to end of line) to clear any leftover content in the margin column and any stale characters past the last painted cell. The renderer emits the entire frame in a single ordered `Terminal::flush()` write-loop on `STDOUT_FILENO` (the diagnosis-only `JOT_RENDER_CHUNK_BYTES` chunked path is preserved) so the output is strictly byte-ordered. The cursor is emitted at the end of the frame; `UI::set_cursor()` / `UI::hide_cursor()` only store state and do not write to the terminal, and the cursor is clamped one cell further inside the margin so the cursor itself is never parked on a right-edge cell either. `UI::flush_cursor()` emits just the cursor state for idle frames (the `!needs_redraw` path in `Editor::render()`). This guarantees that every cell the terminal sees is an intentional paint with no stale fragments or scroll-induced corruption.
  **Scroll safety:** `Terminal::restore_terminal()` re-enables autowrap on exit and resets terminal mouse/focus/paste tracking modes so the host shell regains normal mouse selection. `term->clear()` (real `ESC[2J`) is only called from `UI::invalidate()` (resize / first-frame / theme change); it must not be called from any timer or per-frame path.
- `render/`: drawing of buffer, minimap, overlays, panels, and easter-egg effects. Consumes the grid produced by `ui/`; files are named by surface (`frame.cpp`, `buffer.cpp`, `ui.cpp`, etc.).
- `edit/`: text editing, cursor movement, selection, clipboard, search, sidebar, and discrete editing features (duplicate, join, sort, shuffle, stats, etc.); discrete features use direct action names such as `duplicate.cpp`, not `feature_duplicate.cpp`.
- `input/`: keyboard and mouse dispatch, command palette, command actions, navigation, and modeless editor keyboard handling. The user-facing command layer lives here. Command code is grouped under `commands/`, palette code under `palette/`, and modeless editing dispatch under `modes/modeless.cpp`.
- `features/`: pure syntax helpers (capture mapping and regex highlighter), bracket matching, autoclose, config parsing, and conservative C++ header/source assist helpers. Pure text-manipulation helpers and config loaders. Tree-sitter runtime/catalog/install code lives under `features/tree_sitter/`; editor-bound syntax cache/highlighting glue lives in `core/integrations/`.
- `tools/`: integrated terminal (PTY-backed with libvterm emulation), debugger DAP client, telescope file finder, image viewer, native LSP client, Discord RPC.
  `tools/workspace/search.*` and `tools/symbols/index.*` are pure helper surfaces for workspace text search and regex document-symbol fallback.
- `python_bridge/`: C++ side of the embedded Python bridge. Exposes theme APIs and trusted-local plugin APIs to `src/python/`; plugin support covers user config loading, commands, keymaps, autocmds, buffer/cursor/selection/file helpers, terminal jobs, quick-pick items, and text right-panel providers. Bridge files are named `api*` because the parent directory already carries the Python domain. Keep bindings, lifecycle/plugin logic, and theme methods in normal `.cpp` translation units; do not reintroduce `.inc` implementation chunks included through another `.cpp`.
- `python/`: Python runtime scripts (`jot_api`, theme loader, trusted-local plugin helpers). Bundled, not user-authored.

## Work Guidance

- Match the existing module's style and file layout. New editing primitives go in `edit/`, new UI panels in `render/`, new dispatch logic in `input/`.
- When adding a new C++ source file, register it in the correct module `sources.cmake`. The build will silently miss it otherwise.
- All modules depend on `include/`, `src/`, and the sibling module directories via `jot_configure_object_target`. Do not add ad-hoc include paths.
- The C++17 standard is mandatory. Do not introduce C++20 features.
- Reuse `ui/text.*` for terminal-cell text counting/truncation in UI code instead of adding local `substr`-based clipping helpers. Use `ui/components.*` for simple repeated framed panels and selectable rows when the behavior matches existing layout needs; keep complex editor-specific layouts in `render/` or their owning module.
- The left sidebar is a compact IDE-style activity surface with an Explorer view and a Git source-control view. The activity rail is part of `effective_sidebar_width()` and must be included in render, layout, and hit-testing. Explorer remains a tree with Nerd Font chevrons/icons, cell-aware clipping, an active-file marker, fixed right-side git/diagnostic badges, and a selected-path footer. Git view reuses the existing local Git status/actions and must stay within safe operations: status, diff, staged diff, stage, unstage, and refresh. The sidebar refreshes from disk for external file/folder changes using the existing event-loop timer path and preserves expanded paths, selection, scroll, width, and active view through workspace sessions. Its right border is a mouse-resizable handle; use `effective_sidebar_width()` for layout/hit-testing, persist the width in workspace sessions, and close the sidebar only when an active drag crosses the close threshold. When closed, render only a one-cell left-edge opener handle as an overlay; it must not reserve layout width until clicked or dragged open. Do not render visible shortcut/how-to hints in the explorer chrome.
- The top menu bar is an always-visible one-row application chrome. Keep panes, sidebar, mouse hit-testing, and resize geometry offset below it; menu actions should reuse existing editor commands or direct editor methods instead of duplicating behavior.
- The right panel is a reusable far-right dock for tabbed tools. Reserve its width with `effective_right_panel_width()`, route content through `active_right_panel_tab`, consume mouse input inside its rectangle before editor hit-testing, resize it from its left border, persist `right_panel_width` in workspace sessions, and do not model it as a split pane.
- Split panes use VS Code-like editor-group behavior: creating a split duplicates the active pane's current buffer and local tab list into the new pane, then focuses the new pane. Directional pane focus chooses the nearest pane by geometry; next/previous pane commands remain linear cycling. Pane resize targets the nearest internal split border in the requested direction, and mouse drag resize is limited to internal split borders.
- Pane file tabs are independently scrollable per pane via `SplitPane::tab_scroll_index`. Opening, switching, cycling, restoring, or closing buffers must keep the active local tab visible; mouse wheel and overflow controls on the tab row move the visible tab window without changing the active buffer.
- The startup/no-file editor state uses a hidden placeholder `FileBuffer`. Pristine placeholders do not render pane file tabs and instead show the centered empty-state prompt; explicit new unsaved buffers and modified placeholders are normal unnamed tabs.
- Editor panes reserve the rightmost content column for the vertical scrollbar whenever there is enough pane width. Keep `render_scrollbar()`, `render_buffer_content()`, mouse scrollbar hit-testing, minimap width adjustment, and visible-row scroll clamping in sync.
- Workspace sessions are the project-continuation path. No-argument startup resumes the most recent valid workspace via `recent_workspaces[0]` and `open_workspace(..., true)`; if none exists, it falls back to the home menu. `:home` shows a VS Code-inspired start surface with core actions and folder-first recent entries, and `:resume` re-runs the last-workspace resume flow. Keep per-workspace session files backward compatible unless an explicit migration is planned.
- Smart paste indentation is enabled by `smart_paste_indent=true` and applies only to multi-line editor paste when `auto_indent` is also enabled. It strips common leading indentation from non-empty pasted lines, rebases the block to the cursor line's indentation, adjusts one indent level for obvious opener/closer context, and leaves single-line paste unchanged.
- The search panel is a compact buffer-local find/replace surface. It owns live match state with match lengths, supports case/whole-word/regex modes, current/all replacement, and must keep replacements undoable as a single operation per command.
- Code folding is buffer-local visual state. `features/folding.cpp` owns fold detection, collapsed-range serialization, and visible-line mapping; render, mouse, cursor, scrollbar, and minimap code must use those helpers instead of treating `scroll_offset` as a simple physical-line row whenever folds are active. Folding preserves the folded header's visible row when collapsing a block so the target block does not visually jump away from the user. Folding uses a dedicated gutter indicator beside line numbers, must be clickable, persists collapsed ranges through workspace sessions and the per-file fold-state cache when ranges still match, and must never delete or mutate file text.
- The command palette is a bottom ex-style fuzzy command line with live structured suggestions. Keep entry/toggle state in `input/palette/palette.cpp`, key handling in `input/palette/input.cpp`, suggestions in `input/palette/suggestions.cpp`, ex-command execution in `input/commands/ex.cpp`, generic command dispatch in `input/commands/execute.cpp`, help text in `input/commands/help.cpp`, and menu-bar actions in `input/menu_bar.cpp`. Built-in suggestions carry insert text, label, category, detail, and score. Trusted local Python plugins may register ex commands through the Python bridge; route their execution through `PythonAPI::run_plugin_command()` and show their palette entries under the `Plugin` category. Plugin management commands `:plugins`, `:reloadplugins`, and `:pluginpanel <name>` stay built-in so users can recover from plugin command conflicts.
- Modeless editing is the default user workflow. Typing edits text immediately; selection uses mouse or Shift+Arrow extension; `Esc` clears selection or closes active floating UI; `Ctrl+C`/`Ctrl+X`/`Ctrl+V` are copy/cut/paste. Keep editor keyboard behavior in `input/modes/modeless.cpp` and route global shortcuts through `input/modes/dispatch.cpp`. Trusted local plugin keymaps are checked before normal built-in shortcuts, except emergency quit handling remains reserved.
- Quick-pick is the reusable centered picker for project search, diagnostics, and document symbols. Keep state and behavior in `core/surfaces/quick_pick.cpp`, rendering in `render/ui.cpp`, command entry points in `input/commands/ex.cpp`, and keyboard shortcuts in `input/modes/modeless.cpp`. Project search uses the pure `WorkspaceSearch` helper; document symbols prefer LSP `textDocument/documentSymbol` and fall back to `SymbolIndex` for saved/unsaved buffers.
- Tree-sitter setup commands are visible terminal workflows. `:tsinstall <language-or-github-url>` opens an integrated terminal and builds a parser shared library into the user-local Tree-sitter runtime path; it must not silently run privileged commands. It also opens the `:tsstatus` modal, tracks stable `[jot:treesitter]` progress markers from the install terminal, reloads Tree-sitter after success, and moves the language into the active or installed list. `:tsreload` refreshes parser/query caches, closes loaded parser libraries, and reattempts parser loads without restarting the editor. `:tsstatus` shows a modal grouped by active, installing, installed, and uninstalled catalog languages, with parser/query diagnostics when fallback occurs. User config keys `treesitter_library_paths`, `treesitter_query_paths`, and `treesitter_language_overrides` extend discovery. Header files keep the catalog default `.h` → C mapping unless buffer context indicates C++ (same-stem `.cpp`/`.cc`/`.cxx` sibling or C++-specific tokens); explicit `treesitter_language_overrides` entries for `.h` override that heuristic. C++ highlighting should remain Tree-sitter-backed via built-in or minimal built-in query fallback when an installed runtime query is incompatible.
- Git integration is a safe local workflow surface. The UI may show status, diff, staged diff, blame, log, and may stage, unstage, or commit with an explicit message. Do not add destructive Git operations (discard/reset/checkout) or network operations (push/pull/fetch) to editor commands or context menus without an explicit plan and user approval.
- LSP completion, hover, and definition navigation are native C++ UI/state. Keep completion filtering live while the cursor remains on the same prefix, preserve selection where possible, prefer LSP `textEdit` ranges, and expand LSP snippets conservatively with cursor placement for `$0`/tabstop defaults while leaving full placeholder navigation out of scope. Hover uses a compact popup for both command-triggered and debounced mouse hover, renders fenced Markdown code blocks with syntax colors, must discard stale cursor/pointer results, and must not reveal the cursor or change editor scroll from passive mouse motion. Ctrl-click on an editor symbol requests definition when the terminal reports mouse modifiers. Definition opens cross-file targets as preview tabs, records a bounded back stack, and `:lspback` restores the previous editor location.
- C++ assist commands are intentionally conservative. `:cpppair` creates a matching header/source pair without editing CMake files. `:cppimpl` parses class method declarations from headers, detects already implemented source definitions, creates the counterpart `.cpp` when missing, and appends only generated stubs for missing methods. Keep complex templates/operators/defaulted/deleted/pure-virtual methods skipped unless the parser and tests are expanded together.
- The status bar is a fixed two-row render surface. The first row is file-first, segmented, and width-budgeted; file and cursor context must survive narrow terminals while optional segments drop before overlap. Optional right-side segments may report Git, LSP, syntax engine (`TS <lang>`, `Regex <ext>`, or `Syntax off`), terminals, autosave, theme, RPC, and encoding. The second row is reserved for transient messages or quiet workspace context. Do not add shortcut-hint spam to the status bar.
- Terminal tasks load from local `<workspace>/.jot/tasks.json` plus global `~/.config/jot/configs/tasks.json`; local tasks override global tasks with the same name. The v1 schema is `{ "tasks": { "<name>": "<shell command>" } }`. Task execution uses labeled integrated-terminal tabs and runs local tasks from the workspace root.
- Debugger sessions are native C++ state backed by Debug Adapter Protocol clients in `tools/debugger/client.*`. Local `<workspace>/.jot/debug.json` overrides global `~/.config/jot/configs/debug.json`; the v1 schema is `{ "sessions": { "<name>": { "adapter": "gdb|lldb", "program": "...", "args": [], "cwd": "...", "env": {} } } }`. Live debuggee processes are not restored from workspace sessions.
- Integrated terminal toggles (`Ctrl+\`` plus the existing backtick escape fallbacks) open or focus the bottom panel when hidden/unfocused, and hide it when the active terminal is already focused. `Esc` only releases terminal focus and keeps the panel visible. Terminal startup must not send synthetic input to the shell.
- Integrated terminal styled cells must preserve libvterm default foreground/background markers and resolve them against `Theme::fg_terminal` / `Theme::bg_terminal` during rendering. Do not collapse default terminal colors to fixed xterm indexes before the theme is applied; explicit ANSI colors should remain explicit.
- Output to the terminal is mediated by `ui/terminal.cpp`. Do not `printf`/`std::cout` directly to `stdout` from render code; see `PLANS.md` Fix 2 for the history of this rule.
- The image viewer is a focused full-content surface below the top menu and above the status bar, not a split pane. Clear its owned grid area before positioning terminal graphics, keep the status bar visible, hide the text cursor while active, and delete/redraw real Kitty/Sixel graphics when the view rectangle changes. Real image viewer backends emit terminal graphics escape sequences only after `UI::render()` has flushed the normal grid frame, through `UI::emit_raw_after_frame()` / `Terminal::write()`. Do not interleave Kitty/Sixel image bytes with row painting; the grid renderer would overwrite them or reorder terminal state.
- `term->clear()` (the real terminal `ESC[2J` clear) is only allowed inside `UI::invalidate()`. It must not be called from any timer, periodic redraw, or per-frame path; doing so causes the visible background flicker and stale-overlap bug documented in `PLANS.md`.
- Cursor state is deferred. `UI::set_cursor()` / `UI::hide_cursor()` only store the pending position/visibility; the actual terminal escape sequences are emitted by `UI::render()` (at frame-end) or `UI::flush_cursor()` (for idle frames). Never interleave cursor flushes with a render frame. `UI::clear()` resets the in-memory `grid` only and is safe to call every frame.

## Verification

- Build: `cmake -S . -B build -DCMAKE_BUILD_TYPE=Release` then `cmake --build build -j`. Expect all module `OBJECT` and `STATIC` libraries plus `jot_engine` to compile.
- Run: `./build/jot` opens an empty session; `./build/jot path/to/file` opens a file; `./build/jot path/to/folder` opens a workspace.
- Render regression checks (see `PLANS.md` for the contracts):
  - **Small/partial canvas (width and height):** home menu and editor must fill the real terminal size on first paint, not a tiny centered area. Force sizes via `COLUMNS=120 LINES=40 ./build/jot`, `COLUMNS=200 LINES=80 ./build/jot`, etc. Confirm the UI grid uses the full reported width *and* height. A `grep -oP '\[(\d+);1H'` of the captured output should show the highest row reaching the reported `LINES` value.
  - **Stale grid:** open a large file and confirm all visible text renders on the first frame, and remains visible after several idle seconds.
  - **Flicker/overlap:** leave jot idle for at least 10 seconds (both home menu and editor) and confirm no background changes, no stale panels or text overlap, and no periodic refresh artifacts. A `grep -c '\[2J'` of the terminal capture over 7+ seconds should return 0 or 1 (the initial alternate-screen clear), not 2+.
  - **Scroll corruption / stale fragments:** idle in a tall terminal for 10s; no `\n`, `\x1bD`, or `\x1bM` should appear. The byte-level capture must show that every row is painted from column 0 (via `\x1b[y;1H` move-cursor sequences). Rendered rows must be padded to full width minus the render margin — no partial cells from a previous frame can survive.
  - **Right-edge safety margin:** the renderer leaves exactly the rightmost physical column of every row untouched. Wider margins are not supported so the visible right border always lands on the last paintable column instead of a wider blank strip. The cursor is clamped one cell further inside. The previous right-edge safety behavior remains: no fullscreen/large-width border corruption, no wrapping drift, and no missing right border.
  - **Per-row clear and non-blocking drain:** every painted row is followed by `ESC[K` (erase to end of line) and a `try_drain()` call. Verify in the raw capture that there is exactly one `ESC[K` per row. The `try_drain()` is non-blocking and pushes accumulated output to the kernel as the frame is built; the `bytes=…` in the `FRAME` marker reflects only the last `flush()` at frame end, which is the leftover after the in-frame drains. The `JOT_RENDER_CHUNK_BYTES=<n>` env var controls the drain threshold (default 4096). Typing must remain responsive at 20+ keys/sec with no freeze at any terminal size.
  - **Large-file typing responsiveness:** open a 5M-line file (or run `python3 -c "for i in range(5_000_000): print(i)" > /tmp/big.log` and open it) and type 100 characters. The keystrokes must complete in ~2s at 20 keys/sec, matching the rate of stdin, with no freeze or visible lag. The undo system must capture only a window of lines around the cursor, not the full 5M-line buffer, on every keystroke.
